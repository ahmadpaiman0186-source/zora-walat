## Phase 6 — Transaction path latency (observations, no invented SLA numbers)

- **Pricing quote:** In-process `resolveCheckoutPricing` + JSON; no Stripe round-trip — suitable for fast, deterministic UI preview (verified by `verify:local-pricing` and HTTP doctor probe).
- **Checkout session creation:** Uses Stripe API only (`checkout.sessions.create`); fulfillment is **not** awaited in that handler.
- **Webhook:** Persists idempotently in a DB transaction, then **schedules** fulfillment (`scheduleFulfillmentProcessing`) and fee capture after commit; responds `{ received: true }` after post-commit scheduling (see `stripeWebhook.routes.js`). Webhook handler still does **synchronous DB work** before HTTP 200 — latency tied to DB + transaction scope, not to external airtime API in the same request (fulfillment queue is separate).

**No end-to-end latency KPI** was measured in this session beyond integration test logs (lab-only webhook timing appears inside integration output; treat as dev harness, not production SLO).
